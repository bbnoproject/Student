# 작업 인수인계 문서

이 문서는 폴더를 전달받은 사람이 이전 작업 의도와 현재 구조를 안정적으로 이어받기 위한 컨텍스트입니다.

## 1. 프로젝트 목적

이 프로젝트는 게임 기획 과정 학생들의 원본 데이터를 모아 로컬 대시보드에서 학생별 성장, 상태, 출결, 프로젝트, 면담, 취업 준비 흐름을 확인하기 위한 작업입니다.

핵심 목적은 단순 점수화가 아니라 운영자가 학생의 현재 상태를 빠르게 파악하고, 왜 그런 상태로 판단되었는지 근거를 함께 확인하는 것입니다.

## 2. 현재 산출 구조

- `Data/`
  원본 데이터 폴더입니다. 엑셀, 마크다운, HTML, CSV, 이미지 자료가 섞여 있습니다.

- `scripts/build_app_data.py`
  `Data/` 원본을 읽어 `App/data.js`를 생성합니다. 앱 화면에서 직접 사용하는 통합 데이터가 여기서 만들어집니다.

- `scripts/reprocess_student_data.py`
  `App/data.js`를 바탕으로 `ProcessedData/` 아래 학생별 JSON/Markdown 산출물을 생성합니다.

- `App/`
  정적 웹 프로토타입입니다. 별도 빌드 없이 `App/index.html`에서 메인 로비를 볼 수 있고, 학생 카드를 누르면 `App/student.html?id=...`로 이동합니다.

- `ProcessedData/`
  학생별 재가공 결과입니다. 사람이 직접 참고할 수 있는 분석 산출물이지만, 원본처럼 취급하지 마세요. 필요하면 스크립트로 다시 생성하는 파생 데이터입니다.

## 3. 데이터 흐름

기본 흐름은 아래와 같습니다.

```text
Data
  -> scripts/build_app_data.py
  -> App/data.js
  -> scripts/reprocess_student_data.py
  -> ProcessedData
```

원본 데이터가 변경되면 `build_app_data.py`를 먼저 실행하고, 그 다음 `reprocess_student_data.py`를 실행합니다.

```powershell
$env:PYTHONIOENCODING='utf-8'
& "C:\Users\KGA_JJ\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe" .\scripts\build_app_data.py
& "C:\Users\KGA_JJ\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe" .\scripts\reprocess_student_data.py
```

다른 PC에서 위 경로가 없으면 설치된 Python으로 아래처럼 실행합니다.

```powershell
$env:PYTHONIOENCODING='utf-8'
python .\scripts\build_app_data.py
python .\scripts\reprocess_student_data.py
```

현재 기준 정상 재가공 결과는 학생 80명입니다. 과거에 `학생 데이터.xlsx` 안의 `이탈자` 구분 행이 학생처럼 들어간 문제가 있었고, 현재는 기본정보가 없는 구분 행을 건너뛰도록 수정되어 있습니다.

## 4. 새로 추가된 원본 데이터의 의미

최근 `Data/`에 다음 폴더가 추가되었습니다.

- `Data/학생 정보/`
  직원 또는 운영진이 학생에 대해 기록한 데이터입니다. 학생 성향, 특징, 한줄평, 관계, 희망 직무, 총평, Good/Bad 기록 등이 들어 있습니다.

- `Data/학생 대원카드/`
  학생들이 모집 단계에서 작성한 자기소개성 자료입니다. 모든 학생에게 있는 것은 아닙니다. 없는 학생도 정상입니다.

현재 파이프라인은 HTML/CSV 텍스트를 읽습니다. 폴더 안의 이미지 파일은 OCR 처리하지 않습니다.

반영 방식:

- `학생 정보`는 학생별 `staffProfile`로 저장됩니다.
- `학생 대원카드`는 학생별 `cadetCard`로 저장됩니다.
- 대원카드의 자기소개, 지원동기, 목표, 관심 분야는 입과 정보와 학습 근거에도 보조적으로 반영됩니다.
- 이탈자 CSV는 `managementStatus = "이탈"` 및 `dropoutInfo`로 반영됩니다.

## 5. 학생 상태 분류 규칙

관리 상태는 현재 세 가지를 사용합니다.

- `일반`
  현재 교육 운영과 관찰 대상에 포함되는 학생입니다.

- `취업`
  취업 또는 채용 연계가 확인되어 진로 관리 중심으로 보는 학생입니다.

- `이탈`
  과정 이탈로 일반 관리 대상에서는 제외하지만 통계에는 표시해야 하는 학생입니다.

중요 규칙:

- 이탈 학생은 로비의 `일반 인원` 필터에서 제외됩니다.
- 이탈 학생은 `과정이탈 인원` 필터에서만 별도로 확인할 수 있어야 합니다.
- 이탈 학생도 전체 통계에는 표시합니다.
- `dropoutInfo`가 있거나 원천 데이터에서 `이탈`로 들어온 학생은 브라우저 localStorage의 예전 수정값이 남아 있어도 항상 과정이탈로 계산합니다.
- `이탈자`는 학생명이 아니라 엑셀 안의 구분 행입니다. 학생으로 생성되면 안 됩니다.

현재 원천 이탈 학생은 다음 5명입니다.

- 김수환
- 김지훈
- 김태훈
- 이길준
- 한가영

## 6. 로비 페이지 작업 의도

로비는 전체 현황을 빠르게 훑는 화면입니다. 과한 설명형 랜딩 페이지가 아니라, 운영자가 바로 학생을 검색하고 필터링하는 작업 화면이어야 합니다.

현재 의도:

- 화이트 모드는 흰 바탕과 파란 강조색을 기준으로 합니다.
- 다크 모드는 같은 파란 강조색 계열과 어울리게 조정합니다.
- 상단 로비는 불필요한 공간을 줄이고 핵심 통계만 보여줍니다.
- 과정이탈자는 일반 관리 대상에서 제외하되, 통계와 별도 필터에서 확인 가능해야 합니다.
- 학생 카드에는 현재 상태와 관리 상태 배지가 함께 보여야 합니다.

관련 파일:

- `App/scripts/lobby.js`
- `App/styles/lobby.css`
- `App/scripts/common.js`

## 7. 학생 개인 페이지 작업 의도

학생 개인 페이지는 학생별 상태와 근거를 확인하는 운영 화면입니다.

현재 의도:

- 학생 정보 수정은 개인 페이지에서 바로 입력하는 것이 아니라 `수정` 버튼을 눌러 수정 화면으로 들어가서 진행합니다.
- 수정값은 현재 브라우저의 localStorage에 저장됩니다. 폴더를 다른 사람에게 주면 이 localStorage 값은 전달되지 않습니다.
- 개인정보 개요는 낭비되는 공간을 줄이고 카드와 폰트를 작고 깔끔하게 유지합니다.
- 점수와 다각형 그래프가 함께 있을 때 공간이 커지지 않도록 레이아웃을 조정합니다.
- 현재 상태, 출결기록 등 요약 카드는 과하게 크게 만들지 않습니다.
- 학생 상태나 능력치를 표시할 때는 이유를 함께 표시합니다.
- 마일스톤은 아직 진행되지 않은 추정치 데이터를 입력하지 않습니다.
- 마일스톤 네이밍은 길게 쓰지 않고 `실습(1차)`처럼 짧게 표시합니다.

관련 파일:

- `App/student.html`
- `App/scripts/student.js`
- `App/styles/student.css`
- `App/styles.css`
- `App/scripts/common.js`

## 8. 평가 준거와 교육학 근거

현재 능력치와 성향 프로파일은 6개 준거로 봅니다.

- 학습 자기조절
- 학습 참여와 지속성
- 협업과 관계 형성
- 도전 대응과 회복탄력성
- 성찰과 피드백 활용
- 진로 목적성과 전문성

중요한 설계 의도:

- 4개 준거만으로는 학생 판단이 불명확하다고 보고 6개 준거로 확장했습니다.
- 각 준거는 단순 점수 설명이 아니라 교육학적 평가 방식을 반영해야 합니다.
- `!` 아이콘을 누르면 해당 준거의 판단기준 팝업이 열립니다.
- 팝업에는 핵심 질문, 주요 데이터, 루브릭 평가, 평정 척도, 판정 유의, 점수 해석이 들어갑니다.

현재 적용한 교육학적 프레임:

- 분석적 루브릭 평가
  하나의 총평으로만 보지 않고 수행 과정, 참여 지속성, 협업 역할, 회복 행동, 성찰, 진로 구체화처럼 하위 준거를 나누어 판단합니다.

- 평정법
  1~4점 척도를 사용합니다.
  - 1점: 관찰과 지원이 시급한 수준
  - 2점: 형성 중인 수준
  - 3점: 기대 수행에 도달한 수준
  - 4점: 전이와 확장이 가능한 수준

학생별 이유 문장은 `긴급 지원`, `형성 중`, `안정`, `확장` 같은 평정 결과와 실제 데이터 근거가 같이 보이도록 작성되어야 합니다.

관련 파일:

- `App/scripts/common.js`
  `PROFILE_CRITERIA`, `profileReason`, `renderProfileReasonBars`

- `App/scripts/student.js`
  `renderCriterionModal`

## 9. 통계와 이탈자 처리 주의사항

이 프로젝트에서 "관리 대상"과 "전체 통계"는 다르게 봅니다.

- 관리 대상: 이탈자를 제외한 학생
- 전체 통계: 이탈자를 포함한 전체 학생

따라서 이탈자를 완전히 숨기면 안 됩니다. 대신 아래처럼 분리해야 합니다.

- 로비 기본 필터: 일반 인원
- 별도 필터: 과정이탈 인원
- 전체 필터: 전체 인원
- 통계 카드: 과정이탈 수 표시

로컬 수정값이 남아 있을 경우에도 원천 이탈 기록이 우선해야 합니다.

## 10. 검증 방법

JS 문법 검사는 아래처럼 합니다.

```powershell
node --check App\scripts\common.js
node --check App\scripts\lobby.js
node --check App\scripts\student.js
```

데이터 재가공 후 확인할 값:

- 전체 학생 수: 80명
- 과정이탈 학생 수: 5명
- `이탈자`라는 학생 폴더가 없어야 합니다.
- `App/data.js`에 `staffProfile`, `cadetCard`, `dropoutInfo`가 포함되어야 합니다.
- 대원카드가 없는 학생이 있어도 오류가 나면 안 됩니다.

간단한 데이터 확인 예:

```powershell
node -e "const fs=require('fs'); const text=fs.readFileSync('App/data.js','utf8'); console.log(text.includes('dropoutInfo'), text.includes('cadetCard'), text.includes('staffProfile'))"
```

## 11. 브라우저와 localStorage 주의

학생 수정 화면에서 바꾼 값은 브라우저 localStorage에 저장됩니다.

이 값은 다음에 포함되지 않습니다.

- `Data/`
- `App/data.js`
- `ProcessedData/`
- Git 또는 폴더 압축본

따라서 다른 사람에게 폴더를 넘기면 로컬 수정값은 전달되지 않습니다. 운영상 공유가 필요하면 이후 localStorage 값을 내보내는 기능이나 원본 데이터 반영 절차를 별도로 만들어야 합니다.

## 12. 다음 작업자가 조심할 점

- `ProcessedData/`는 파생 산출물이므로 원천 수정은 `Data/` 또는 스크립트에서 해야 합니다.
- 원본 데이터 폴더에 HTML/CSV/이미지가 섞여 있습니다. 현재 이미지는 OCR하지 않습니다.
- 이름 매칭은 학생명 기반입니다. 동명이인 또는 `김현우A/B` 같은 구분명이 있으면 조심해야 합니다.
- 엑셀의 구분 행, 제목 행, 빈 행이 학생으로 생성되지 않도록 해야 합니다.
- 이탈자는 일반 관리 대상에서 빠져야 하지만, 통계에서 사라지면 안 됩니다.
- UI는 운영 도구입니다. 마케팅식 랜딩 화면보다 밀도 있고 읽기 쉬운 화면을 우선합니다.
- 루브릭/평정 설명은 학생을 낙인찍는 표현이 아니라 교육적 지원 판단으로 읽히게 유지합니다.

## 13. 현재 주요 변경 요약

- `Data/학생 정보`, `Data/학생 대원카드`를 데이터 재가공 파이프라인에 연결했습니다.
- 학생별 `staffProfile`, `cadetCard`, `dropoutInfo`가 생성됩니다.
- 이탈자 CSV 기반으로 과정이탈 상태를 반영합니다.
- 가짜 학생 `이탈자`가 생성되지 않도록 기본정보 없는 구분 행을 제외했습니다.
- 로비 페이지에 `일반 인원`, `과정이탈 인원`, `전체 인원` 필터를 구성했습니다.
- 로비 상단 공간을 줄이고 핵심 관리 통계 중심으로 정리했습니다.
- 학생 개인 페이지에 정보 수정 흐름과 관리 상태 표시를 추가했습니다.
- 현재 능력치/성향 판단에 이유 문장과 `!` 판단기준 팝업을 추가했습니다.
- 평가 준거에 루브릭 평가와 평정법 기반 설명을 보강했습니다.
- 마일스톤 표기를 `실습(1차)`처럼 짧게 정리했고, 아직 진행되지 않은 추정치는 넣지 않도록 했습니다.

## 14. 추천 인수인계 순서

새 작업자는 아래 순서로 보면 됩니다.

1. `README.md`
2. `HANDOFF.md`
3. `App/index.html`

이 순서로 보면 프로젝트 목적, 데이터 생성 흐름, 화면 동작, 판단 기준을 자연스럽게 이어받을 수 있습니다.
