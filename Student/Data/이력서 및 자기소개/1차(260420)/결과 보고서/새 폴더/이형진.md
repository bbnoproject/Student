# 이형진_자기소개서

## 종합 판단
개발자 친화적 시스템 기획자라는 캐릭터가 선명하고, 기술 병목 대응과 제작 파이프라인 개선 사례가 강합니다. 시스템 기획 직무에 맞는 어필이 일관적입니다.

## 잘 한 부분
- “기획 리뷰와 Q&A 세션”을 진행한 이유가 분명합니다. 문서가 다르게 해석될 수 있기 때문에 개발팀과 이해도를 맞췄다는 점이 좋습니다.
- A* 길찾기 알고리즘 구현 한계 상황에서 기획을 고집하지 않고 기능을 단순화한 점은 실무적입니다. 핵심 재미와 구현 가능성을 함께 고려했습니다.
- “A안과 B안으로 분리된 기획서”를 준비한 점은 리스크 대응 역량을 보여줍니다.
- HTML 맵 에디터와 `.md` 데이터 추출 파이프라인은 제작 효율을 높인 강한 행동 근거입니다.

## 개선 방향 제안
- 문서가 다소 성과 나열형입니다. 왜 시스템 기획에 끌렸는지, 어떤 경험에서 본인의 기획 철학이 생겼는지 감정선이 조금 더 설명되면 좋습니다.
- 학원 프로젝트의 출시, 유저 수, 내부 투표 결과 같은 지표는 참고가 될 수 있지만, 현업 기준에서는 크지 않은 근거로 읽힐 수 있습니다. 지금 문서에서는 그 숫자보다 기획 리뷰, 대안 설계, 병목 대응 과정을 더 앞세우는 편이 좋습니다.
- A안과 B안의 구체 차이가 나오면 더 좋습니다. 어떤 기능을 유지하고 어떤 기능을 축소했는지 설명해보는 방향을 제안합니다.
- “기본기”라는 표현이 반복됩니다. 기본기를 무엇으로 정의하는지, 문서 구조나 데이터 설계를 볼 때 어떤 기준을 중요하게 여기는지 더 보여주면 좋습니다.
