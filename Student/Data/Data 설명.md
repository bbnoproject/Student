# 학생 데이터 설명

## 1. 문서 목적

이 문서는 `Data` 폴더에 포함된 학생 운영 데이터를 실제 파일 기준으로 설명하기 위한 문서입니다.  
단순히 파일 이름만 정리하는 것이 아니라, 아래 내용을 함께 기록합니다.

- 어떤 파일이 어떤 목적의 데이터인지
- 원본 입력 데이터와 운영용 가공 데이터가 어떻게 나뉘는지
- 사이트에서 어떤 카테고리로 흡수할 수 있는지
- 정규화 또는 구조 통일이 필요한 지점이 무엇인지

현재 `Data` 폴더에는 아래와 같은 형태의 데이터가 함께 존재합니다.

- 엑셀 기반 정형 데이터 `9개`
- 프로젝트 협업 구조를 담은 Markdown 문서 `1개`
- 학생별 면담 Markdown 문서 `75개`
- 취업 준비 문서/피드백 텍스트 및 Markdown 폴더 `1개`

즉, 이 폴더는 `정형 데이터 + 반정형 운영 문서 + 비정형 텍스트 데이터`가 혼합된 상태입니다.

## 2. 데이터 분류

사이트 기획 기준으로 보면 현재 데이터는 아래 5개 영역으로 나뉩니다.

### A. 학생 마스터 데이터

- `학생 데이터.xlsx`

### B. 모집단계 데이터

- `모집 결과.xlsx`

### C. 교과 과정 운영 데이터

- `출결 기입 시트.xlsx`
- `수업 시간표.xlsx`
- `1차 프로젝트 데일리 체크인.xlsx`
- `2차 프로젝트 데일리 체크인.xlsx`
- `3차 프로젝트 데일리 체크인.xlsx`
- `1차 프로젝트 회고.xlsx`
- `2차 프로젝트 회고.xlsx`
- `프로젝트 팀 구성.md`

### D. 취업 문서 학습 데이터

- `이력서 및 자기소개/**/*`

### E. 면담 및 상담 기록

- `면담 기록/*.md`

## 4. 상세 설명

### 4.1 학생 데이터.xlsx

#### 개요

학생 개인 기본 정보가 담긴 마스터 파일입니다.  
현재 확인된 시트는 `기획4기` 1개입니다.

#### 주요 컬럼

- 이름
- 전화번호
- 성별
- 생년월일
- 최종학력
- 현업경력
- 실주소지
- 특이사항
- 결석 관련 메모

#### 특성

- 학생 식별을 위한 핵심 정보가 가장 많이 모여 있는 파일입니다.
- 전화번호, 생년월일, 주소 등 민감 정보가 포함되어 있습니다.
- 실사용 헤더는 앞부분에 집중되어 있고, 뒤쪽 열은 비어 있거나 사용 흔적이 약한 편입니다.

#### 사이트 반영 방향

- 학생 기본 프로필의 기준 데이터로 사용
- 학생 ID 생성 전, `이름 + 전화번호` 또는 `이름 + 생년월일` 기반 매칭 후보로 활용
- 민감 정보 접근 권한을 별도로 설계해야 함

---

### 4.2 모집 결과.xlsx

#### 개요

학생 모집 및 면접 단계 데이터를 담고 있는 파일입니다.  
현재 확인된 시트는 아래 3개입니다.

- `설문지 응답 시트1`
- `면접활용시트`
- `면접 결과 관리`

#### 시트별 설명

##### 1) 설문지 응답 시트1

지원서 원본 응답 데이터입니다.  
실제 입력 원본으로 보는 것이 적절합니다.

확인된 주요 컬럼 예시

- 이메일 주소
- 이름
- 성별
- 나이
- 최종학력
- 전공 여부
- 학교와 전공
- 1분 자기소개
- 과정 신청 이유
- 게임 관련 학습/업무 경험
- 관련 취업 분야 이해도
- 수료 후 목표
- 캠/마이크 보유 여부
- 지병/질환 여부 및 설명
- 갈등 대처 방식
- 실수/실패 대응 방식
- 타인의 부족한 부분에 대한 대응
- 경제적 어려움 여부

##### 2) 면접활용시트

면접 진행 시 사용하는 운영용 시트입니다.

포함 내용 예시

- 지원자 기본 정보 표시 영역
- 평가자 정보
- 질문/안내 리스트
- 평가 기준
- 면접 증빙 관련 안내
- 외부 드라이브 링크

이 시트는 `데이터 원본`이라기보다 `면접 운영 템플릿/작업 시트` 성격이 강합니다.

##### 3) 면접 결과 관리

면접 결과를 정리하고 요약하는 시트입니다.

포함 내용 예시

- 점수 구간별 분류 기준 `A~D`
- 연령대 분류 기준
- 성별 분포
- 면접 결과 리스트

면접 결과 리스트 영역에서 확인된 주요 필드 예시

- 번호
- 이름
- 성별
- 나이
- 학력
- 점수
- 결과
- 총평

#### 사이트 반영 방향

- `설문지 응답 시트1`은 모집단계 원본 데이터로 저장
- `면접 결과 관리`는 면접 결과 데이터와 요약 통계의 참고 소스로 사용
- `면접활용시트`는 사이트 내 면접 입력/조회 UI 설계 참고 자료로 활용

#### 주의사항

- 모집 원본에는 건강 상태, 경제 사정 등 민감도가 높은 정보가 포함되어 있습니다.
- 사이트 반영 시 기본 정보와 민감 설문 항목의 접근 권한을 분리하는 것이 좋습니다.

### 4.3 출결 기입 시트.xlsx

#### 개요

출결과 휴가/공가 운영을 기록하는 파일입니다.  
기간별 시트와 집계 시트가 함께 존재합니다.

확인된 주요 시트 예시

- `12.8~1.7`
- `1.8~2.7`
- `2.8~3.7`
- `3.8~4.8`
- `4.9~5.8`
- `5.9~6.8`
- `6.9~7.8`
- `지각`
- `결석`

#### 구조 설명

##### 1) 기간별 시트

기간별 시트는 학생명/전화번호를 행 기준으로 두고, 날짜를 열 기준으로 두는 매트릭스 구조입니다.

상단에 확인되는 출결 코드

- 결석
- 조퇴
- 지각
- 외출
- 공가
- 휴가

실질 헤더 구조

- 이름
- 전화번호
- 날짜별 출결 상태 셀

셀 안에는 단순 공란뿐 아니라 아래와 같은 메모성 텍스트가 들어갈 수 있습니다.

- 지각
- 결석
- 휴가
- 지각 (사유 포함)

##### 2) 지각 / 결석 시트

학생별 누적 집계 성격의 시트입니다.

확인된 주요 필드 예시

- 이름
- 전화번호
- 총 사용
- 기간별 사용 수
- 휴가 사용일수

#### 사이트 반영 방향

- 기간별 시트는 `일자별 출결 이벤트 원본`으로 파싱
- `지각`, `결석` 시트는 사이트 내 집계 결과 검증용 참고 데이터로 활용
- 장기적으로는 셀 기반 매트릭스 대신 `학생 1명 x 날짜 1건` 구조로 정규화하는 것이 바람직

#### 주의사항

- 현재 파일은 사람이 직접 운영하는 매트릭스 시트 형태라 자동 처리 난도가 높습니다.
- 일부 시트는 템플릿 복제 흔적이나 날짜 검증이 필요한 구간이 있을 수 있으므로, 가져오기 전에 구조 검증 단계가 필요합니다.

### 4.4 수업 시간표.xlsx

#### 개요

과정 운영 일정과 커리큘럼 관련 참고 자료입니다.  
학생 개인 데이터라기보다 과정 메타데이터 성격이 강합니다.

확인된 주요 시트

- `시간표`
- `시간표 조정 안건 양윤석`
- `시트3`
- `시트4`

#### 시트별 설명

##### 1) 시간표

과정의 기준 일정 시트입니다.

확인된 항목 예시

- 개강일
- 종강일
- NO
- 날짜
- 요일
- 교과목(및 능력단위)코드
- 교육 타이틀

##### 2) 시간표 조정 안건 양윤석

기존 시간표와 조정안을 비교하는 운영 검토용 시트입니다.

확인된 항목 예시

- 교과목
- 시수
- 변화(증감)

##### 3) 시트3

기수 회고나 운영 메모 성격의 시트입니다.  
정형 데이터보다는 참고 메모에 가깝습니다.

##### 4) 시트4

과목별 실습 제출/미제출, 평가 미제출자 등을 요약한 운영 시트입니다.

#### 사이트 반영 방향

- 학생 관리 사이트의 핵심 학생 데이터보다는 운영 참고 데이터로 사용
- 추후에는 `과정 메타 정보`, `과제/평가 일정`, `제출 현황 기준표` 쪽으로 분리 가능

### 4.5 1차 프로젝트 데일리 체크인.xlsx

#### 개요

1차 프로젝트 기간 동안 학생들이 작성한 일일 체크인 데이터입니다.

확인된 주요 시트

- `설문지 응답 시트1`
- `종합`
- 일자별 시트 `0120`, `0121`, `0122`, `0123`

#### 시트별 설명

##### 1) 설문지 응답 시트1

원본 응답 데이터입니다.

확인된 컬럼

- 타임스탬프
- 이메일 주소
- 이름
- 소속 팀
- 작성 날짜
- 오늘의 작업량
- 자유로운 한마디

##### 2) 종합

운영진이 보는 집계 시트입니다.

확인된 핵심 항목

- 팀
- 이름
- 미작성 횟수
- 응답 비율
- 날짜별 제출 여부

##### 3) 일자별 시트

운영용 가공 시트입니다.

확인된 컬럼

- 팀
- 이름
- 작업량
- 한마디
- 비고
- 상태

#### 사이트 반영 방향

- 원본 응답은 `프로젝트 일일 기록 원본`
- 종합 시트는 `제출 여부/응답률 검증용`
- 일자별 시트는 `운영 피드백/상태관리` 참고 데이터

### 4.6 2차 프로젝트 데일리 체크인.xlsx

#### 개요

2차 프로젝트의 일일 체크인 데이터입니다.  
1차보다 운영 보조 시트가 늘어났고, 포인트 지급 시트도 포함되어 있습니다.

확인된 주요 시트

- `설문지 응답 시트1`
- `Base`
- `종합`
- `Date`
- `시트4`
- 일자별 시트 `213`, `219`, `220`, `223`, `224`, `225`, `226`, `227`
- `포인트 지급`

#### 시트별 설명

##### 1) 설문지 응답 시트1

원본 응답 데이터입니다.

확인된 컬럼

- 타임스탬프
- 이메일 주소
- 이름
- 소속 팀
- 작성 날짜
- 오늘의 작업 이력
- 자유로운 한마디

1차와 비교하면 `작업량` 중심에서 `작업 이력 서술` 중심으로 바뀐 점이 특징입니다.

##### 2) Base / Date

일자별 시트 생성 및 관리용 보조 시트로 보입니다.

##### 3) 종합

제출 여부와 응답률 집계 시트입니다.

확인된 핵심 항목

- 팀
- 이름
- 미작성 횟수
- 응답 비율
- 날짜별 제출 여부

##### 4) 일자별 시트

운영진 확인용 가공 시트입니다.

확인된 컬럼

- 팀
- 이름
- 작업량
- 한마디
- 비고
- 상태

##### 5) 포인트 지급

프로젝트 보상/운영 이벤트 관리 시트입니다.

확인된 컬럼

- 팀
- 이름
- 지급 포인트
- 지급 여부
- 팀장
- 발표자
- 프로젝트상/특별상 여부

#### 사이트 반영 방향

- 프로젝트 기록 데이터 외에 `운영 보상 이력`이라는 별도 도메인으로 확장 가능
- 현재 기획 범위에서는 우선 참고 데이터로만 관리해도 무방

### 4.7 3차 프로젝트 데일리 체크인.xlsx

#### 개요

3차 프로젝트 진행 중 사용 중인 일일 체크인 파일입니다.  
현재 가장 운영형 구조가 명확하게 잡혀 있는 파일입니다.

확인된 주요 시트

- `Base`
- `Date`
- `설문지 응답 시트1`
- `종합`
- `양식`
- 일자별 시트 `0318` ~ `0408`

#### 시트별 설명

##### 1) 설문지 응답 시트1

원본 응답 데이터입니다.

확인된 컬럼

- 타임스탬프
- 이름
- 소속팀
- 작성 날짜
- 오늘 작업한 내용
- 자유로운 한마디

1차, 2차와 비교했을 때 다음 차이가 있습니다.

- 이메일 주소 컬럼이 사라짐
- 자유 서술형 작업 내용 중심으로 변경됨

##### 2) Base / Date / 양식

일자별 체크인 시트와 양식을 관리하기 위한 보조 시트입니다.

##### 3) 종합

응답률, 미작성 횟수, 날짜별 제출 여부를 관리하는 시트입니다.

##### 4) 일자별 시트

운영진 가공 시트입니다.

확인된 컬럼

- 팀
- 이름
- 작업량
- 한마디
- 비고
- 상태

#### 사이트 반영 방향

- 3차 파일은 현재 진행형 데이터이므로 추후에도 일자별 레코드가 계속 추가될 수 있음
- 사이트 구조에서는 `프로젝트 회차`, `날짜`, `학생`, `제출 시각`, `제출 여부`, `시간 내 제출 여부`, `작업 내용`, `운영 메모`, `상태`로 정규화하는 것이 적절

### 4.10 면담 기록 폴더

#### 개요

학생별 Markdown 문서로 관리되는 면담 기록 폴더입니다.  
현재 학생명 기준 파일 `75개`가 존재합니다.

#### 구조 특징

각 파일은 대체로 아래 구조를 따릅니다.

```md
# {학생명} 면담 기록

## {면담 회차 또는 면담 유형}
- 면담자 : ...
- 면담일 : YYYY-MM-DD

### 내용
...
```

#### 실제 확인된 면담 유형 예시

- 1차 정기 면담
- 2차 정기 면담
- 이탈 면담
- 진로 상담
- 개인 고민 상담
- 프로젝트 관련 고민 상담
- 취업
- 질문
- 기업협약 프로젝트

#### 데이터 특성

- 총 `##` 레벨 섹션이 202개 확인되어, 학생 1명당 복수 면담 기록이 들어 있는 구조입니다.
- 파일명은 학생명 기준입니다.
- 면담자, 면담일, 내용은 비교적 공통적이지만, 세부 소제목은 문서마다 다릅니다.
- 정기면담 양식도 시기나 면담자에 따라 질문 구조가 조금씩 다릅니다.
- 매우 짧은 문서도 존재하며, 예를 들어 `김시온.md`는 사실상 제목만 있는 수준입니다.

#### 사이트 반영 방향

- `학생 1명 : 면담 기록 N건` 구조로 저장
- 공통 메타데이터로 아래 필드를 우선 분리

기본 메타데이터 후보

- 학생명
- 면담 유형
- 면담자
- 면담일
- 본문
- 태그

#### 주의사항

- 이 폴더는 정형화 정도가 낮아 자동 파싱 시 예외 처리가 필요합니다.
- 면담 유형 명칭을 정규화하지 않으면 검색과 통계가 어려워집니다.

### 4.11 프로젝트 팀 구성.md

#### 개요

프로젝트 회차별 팀 구성과 학생 역할을 Markdown으로 정리한 파일입니다.  
엑셀 응답 데이터가 아닌 `협업 구조 메타데이터`라는 점에서 의미가 큽니다.

#### 구조 특징

문서는 아래 흐름으로 구성됩니다.

```md
## 1차 프로젝트

### 1팀
- 학생명 (팀장)
- 학생명
```

핵심적으로 추출 가능한 정보는 아래와 같습니다.

- 프로젝트 회차
- 팀 번호
- 학생 역할
- 함께한 팀원 목록

#### 사이트 반영 방향

- `학생 1명 : 프로젝트 배치 N건` 구조로 저장
- 프로젝트 회고, 데일리 체크인과 같은 회차 기준으로 묶어서 조회
- `팀장 경험`, `PM 경험`, `반복 협업 팀원`, `특정 프로젝트 팀 맥락`을 보조 지표로 활용

#### 활용 가치

- 프로젝트 회고의 협업 서술을 실제 팀 구성과 함께 읽을 수 있습니다.
- 같은 팀원을 여러 번 만난 학생, 특정 역할을 반복 수행한 학생을 파악할 수 있습니다.
- 협업/관계 형성 준거를 해석할 때 `텍스트만이 아닌 실제 팀 맥락`을 함께 볼 수 있습니다.

### 4.12 이력서 및 자기소개 폴더

#### 개요

과정 중 진행된 취업 문서 학습 산출물 폴더입니다.  
1차와 2차에 걸쳐 자기소개서와 이력서 작성을 연습했고, 일부 학생은 피드백과 수정 이력을 남겼습니다.

#### 내부 구조

- `1차(260420)`
  - `자소서 모음집/*.txt`, `*.pdf`
  - `개인 피드백/*.md`
  - `평가 기준.md`, `작성 형식.md`, `피드백 지시서.md`
- `2차(260503)`
  - `피드백/_추출텍스트/*.txt`
  - `피드백/*.md`
  - `학생문서/*.pdf`
  - `이력서 평가 기준.md`, `자기소개서 평가 기준.md`

#### 데이터 성격

- `학생이 직접 작성한 문서`: 진로 목적성, 직무 이해, 자기표현 방식의 근거
- `운영진 피드백`: 형성평가와 수정 방향의 근거
- `평가 기준/작성 형식 문서`: 운영 기준과 문서 과제의 학습 목표를 보여주는 메타자료

#### 사이트 반영 방향

- `학생 1명 : 문서 회차 N건` 구조로 저장
- 문서 종류를 `selfIntroduction`, `resume`, `feedback`으로 분리
- `제출 여부`, `회차`, `수정 여부`, `최신 문서일`을 요약 메타데이터로 함께 저장
- 진로 목적성, 성찰, 자기조절 준거와 연결되는 보조 데이터로 활용

#### 주의사항

- 필수 과제가 아니므로 `미제출 = 부정적 평가`로 바로 해석하면 안 됩니다.
- PDF 원본과 추출 텍스트가 함께 있어, 화면 반영용으로는 텍스트 기반 구조화가 유리합니다.
- 파일명 편차가 일부 있으므로 이름 매칭 규칙을 유연하게 설계해야 합니다.

## 5. 공통 데이터 특성

현재 `Data` 폴더의 데이터는 아래 특징을 공통적으로 가집니다.

### 5.1 원본 데이터와 운영용 가공 시트가 섞여 있음

예시

- `설문지 응답 시트1`: 원본 응답
- `종합`, `Base`, `Date`, `양식`, `포인트 지급`: 운영 및 가공 시트
- `면접활용시트`: 입력/평가용 운영 시트

따라서 사이트에 가져올 때는 `원본`, `운영 가공`, `참고/메타`를 구분해야 합니다.

### 5.2 회차별/시기별로 컬럼이 달라짐

예시

- 프로젝트 데일리 체크인 1차, 2차, 3차의 원본 설문 컬럼이 서로 다름
- 프로젝트 회고 2차에는 팀장 선호도 관련 문항이 추가됨
- 면담 기록의 소제목 체계가 면담자/시점별로 다름

즉, 사이트 설계에서는 `고정 컬럼 + 확장 필드` 구조가 유리합니다.

### 5.3 민감 정보가 다수 포함됨

민감도가 높은 항목 예시

- 전화번호
- 생년월일
- 실주소지
- 건강/지병 정보
- 경제적 어려움 여부
- 상담 내용

이 데이터들은 권한 통제와 마스킹 설계가 필요합니다.

### 5.4 학생 식별 키가 일관되지 않음

어떤 파일은 이메일이 있고, 어떤 파일은 없습니다.  
어떤 파일은 전화번호가 있고, 어떤 파일은 팀/이름 중심입니다.

초기 정합성 확보를 위해 아래 우선순위의 식별 전략이 필요합니다.

1. 내부 `studentId`
2. 이름 + 전화번호
3. 이름 + 생년월일
4. 이름 + 이메일

## 7. 정리

현재 `Data` 폴더는 이미 사이트의 핵심 기능을 설계할 수 있을 정도로 데이터 범위가 넓습니다.  
다만 그대로 가져오기보다는 아래 원칙으로 재구성하는 것이 좋습니다.

1. 원본 시트와 운영용 가공 시트를 분리한다.
2. 학생 공통 식별 체계를 먼저 만든다.
3. 프로젝트/면담처럼 회차별로 형식이 달라지는 데이터는 확장형 구조로 저장한다.
4. 민감 정보는 기본 정보와 분리해 권한 통제를 강화한다.
5. Markdown 면담 기록은 메타데이터와 본문을 분리해 저장한다.

이 문서는 이후 `학생 사이트 데이터 모델`, `파일 업로드 정책`, `Firebase 컬렉션 설계`의 기준 문서로 사용할 수 있습니다.
